# ReceiptAnalyzer — プロジェクト目的（Purpose）

> 目的＝根。SDD（幹）以下すべての出発点。これが揃って初めて NFR・I/F・EDD が書ける。
> 確定日：2026-06-21 / UKIA との対話より

---

## 1. 誰の・何を解決するか

**主な対象**
企業規模を問わず、**チーム単位で会計システムへの証憑手入力が発生している組織**。
個人事業主・小規模事業者から、大企業の一部署・チームまで。共通点は「証憑の手入力が件数に比例して負担になっている」こと。

**解決する課題**
外部から来る証憑（**レシート＝経費／請求書＝債務支払い**）を会計システムへ手入力する作業が、時間とミスを生む。
これを「証憑画像 → 中間データ → 各社の会計システムへの取り込み」へ半自動化し、**入力作業を確認作業へ置き換える**。

**核心の一文**
> 企業規模を問わず、チーム単位で発生する会計システムへの証憑手入力（経費・債務支払い）を、低コストで半自動化する。
> 出力先の会計システム（MF/freee/弥生/独自）と自動化の深さ（CSV取り込み〜API直接登録）は、導入先の事情と承認フロー＝可逆性に応じて**カスタマイズ可能な枝葉**とし、共通の幹（証憑の読み取り→中間データ→人の確認）を製品の核とする。

## アーキテクチャ原則：幹と枝葉（樹木モデル）

```
[入力源] スマホ撮影 / 複合機スキャン
  → [ストレージ] Google Drive / Box / 所定フォルダ          ← 枝葉(可変)
  → [トリガー]   自動検知 or 担当者が手動起動               ← 枝葉(可変)
  ───────────── 取り込み ─────────────
  → [OCR]        Google Vision / Bedrock / ローカル         ← 枝葉(可変・入力エンジン)
  → [構造化AI + 過去データ参照] 文字起こし→勘定科目を振る   ← 幹(核)
  → ★[正本中間JSON] schema_version・証憑種別・必須項目      ← 幹(ピボット＝唯一の正本)
  → [人の確認]                                             ← 幹(不変・責任)
  ───────────── 着地 ─────────────
  → [出力アダプタ] Excel / CSV / API登録(承認あり) / API直接 ← 枝葉(可変・自動化の深さ)
  → [層B] 証憑画像 / PDF を添付（任意）                     ← 枝葉(可変)
```

- **幹（共通・作者が提供）**：証憑読み取り → 正本中間スキーマ → 人の確認
- **枝葉（換装可能・導入先ごと）**：① 入力（ストレージ/トリガー/OCR/LLM）② 出力（会計システム）③ 自動化の深さ ＋ 層B画像添付
- カスタマイズの担い手＝作者または導入企業（共通幹＋参照アダプタ＝半サービス）

## 2. 成功とは何か

**北極星指標：一発合格率**（人が修正せず確認だけで通る率）。数字は仮置き、50〜100枚の eval harness で確定する。

| 層 | 指標 | 仮目標 |
|----|------|--------|
| 幹 | 合計金額・税額（致命） | 99%以上 |
| 幹 | 日付（重要）/ 取引先（重要・表記揺れ許容） | 95% / 90% |
| 枝葉 | CSV取り込みエラーゼロ / API登録成功 | 100% |
| 枝葉 | 勘定科目の提案正解率 | 90% |
| 運用 | 一発合格率（★北極星） | 80%以上 |
| 運用 | 処理時間（GPUなし） / コスト | 2〜8秒 / 無料枠内 |
| 安全 | 可逆性ゲートが機能（不可逆無検証着地ゼロ） | 100% |

**一発合格の定義**：致命項目（金額・税）が正しい AND 重要項目が許容内。全項目完全一致は要求しない。
弱いフィールドは**履歴参照による名寄せ・正規化（幹）**で持ち上げる。

## 3. 非目標（やらないこと）

1. ❌ 可逆性の担保なしに人間（または承認フロー）を抜くこと。※フルオート自体は否定しない／不可逆な無検証着地のみを否定
2. ❌ 会計ソフトそのものの代替（記帳・決算・申告は範囲外。入力の半自動化まで）
3. ❌ 税務・会計上の正当性の最終保証（勘定科目は"提案"、最終責任は人間／専門家）
4. ❌ 会計入力に直結しない文書への汎用OCR（対象はレシート・請求書に限定）
5. ❌ 証憑データを第三者に預ける集中型クラウドSaaS化（自己ホスト型は可／外部預託は不可。層Bの着地先添付は否定しない）
6. ❌ 全会計システムのアダプタを作者が網羅・恒久保守すること（作者は幹＋MF参照実装まで／独自は委譲・共同）
7. ❌ 電子帳簿保存法の正式アーカイブそのもの（壊さない・対応システムへ渡すのみ）／証憑の真正性検証・不正検知

## 4. 制約（不変 / 可変）

| 不変（常に守る） | 可変（導入先次第） |
|----------------|------------------|
| 外部送信範囲を制御可能であること | 使うOCR/LLMエンジン（Vision/Bedrock/ローカル…） |
| 可逆性の担保なき不可逆着地をしない | 出力先の会計システム（MF/freee/弥生/独自） |
| データを手元に既定保存しない（保存はopt-in） | 自動化の深さ・閉域/監査要件 |
| 会計制度準拠を壊さない（軽減税率・インボイス・電帳法） | コスト/無料枠の優先度 |
| 正本スキーマを幹とする／人間の最終責任 | GPU有無・ストレージ・トリガー方式・画像添付の有無 |
| — | 入力源（マウント済みドライブ経由 or Drive API 経由）← 導入先の環境次第で換装可能 |

**証憑画像の3層管理**：層A 一時保持（処理中のみ・既定破棄）／層B 引き渡し・添付（着地先への添付・任意）／層C 電帳法アーカイブ（非担当）。
**プライバシー表現**：「完全ローカル」ではなく「**外部送信範囲を制御可能**」と表現する（OCRで画像を外部送信しうるため）。

## v1 スコープ（beachhead）

- **対象顧客**：大企業のチーム（実証済みの痛み・パイロット可能・設計パートナー）
- **機能**：Drive(人別サブフォルダ)/手動トリガー → Vision OCR → 履歴参照で正本JSON → MF API登録 + 証憑添付（人確認後）
- **v1から外す（後段・要件次第）**：閉域ネットワーク対応／ローカルLLM実験
- **MF API調査結果（2026-06-21 確定）**：
  - 仕訳登録：`POST /api/v3/journals` ✅ 対応済み（OAuth 2.0）
  - 証憑添付：`POST /api/v3/vouchers` ✅ Base64エンコードで対応済み
  - 認証：OAuth 2.0 / アクセストークン有効期限1時間（リフレッシュ必要）
  - 仕訳・証憑ともに可逆（削除API あり）→ フルオート着地の前提条件を満たす
- **LLMエンジン**：`LLM_ENGINE` 設定で切り替え可能（claude / openai / 将来: local）。企業ごとに承認済みエンジンが異なるため枝葉として設計
- **勘定科目判定**：過去仕訳CSV（担当者が準備）＋MF勘定科目マスタ（APIから取得）をLLMにコンテキストとして渡す
- **Googleドライブ構成**（v1参照実装）：
  ```
  00_受信トレイ/{人名}/  ← スマホから保存
  01_処理済み/{人名}/    ← JSON生成完了後に移動
  02_エラー/{人名}/      ← OCR失敗時に移動
  ```
  人名フォルダは提出者識別用。v1では仕訳の `department_id` 等には反映しない（導入先次第で拡張）
