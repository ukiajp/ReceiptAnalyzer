# NFR — 非機能要件（Non-Functional Requirements）

> SDD の「何を作るか」に対し、NFR は「どう動くべきか」を定義する。
> EDD基準のスコアしきい値はここから派生する。
> 依存: `sdd.md` / 確定日: 2026-06-21

---

## 1. パフォーマンス

| 項目 | 要件 | 根拠 |
|---|---|---|
| 1枚あたり処理時間 | **2〜8秒**（GPU不要） | purpose.md 目標値 |
| OCR（Vision API） | 目標 1秒以内 | 実測平均 0.7秒 |
| LLM構造化 | 目標 7秒以内 | クラウドLLM想定 |
| MF API登録+証憑 | 目標 3秒以内 | 2エンドポイント合計 |
| バッチ上限（v1） | 1回あたり50枚まで | 個人用途想定（1〜2枚/日） |

---

## 2. 可用性・信頼性

| 項目 | 要件 |
|---|---|
| 依存サービス障害時の挙動 | OCR失敗 → エラーフォルダへ移動・処理続行（中断しない） |
| MF API登録失敗時 | ログ記録・ファイルは処理済みフォルダに残す（再実行で二重登録しない設計） |
| トークン期限切れ | MF OAuthトークンを自動リフレッシュ（1時間有効期限対応） |
| 再実行安全性 | 同一ファイルを2回処理しない（処理済みフォルダ移動で自然に実現） |

---

## 3. セキュリティ

| 項目 | 要件 |
|---|---|
| 認証情報の管理 | `.env` のみ・コードへのハードコード禁止・`.gitignore` 必須 |
| レシート画像の外部送信 | Vision API（OCR）・LLM API（テキストのみ）に限定 |
| 画像データの保持 | 処理後はDriveに残す（ローカル一時ファイルは削除） |
| MF APIスコープ | `journal.write` + `voucher.write` のみ申請（read不要なら取らない） |
| Google Drive権限 | 特定フォルダIDへのアクセスのみ。Vault全体の権限を与えない |

---

## 4. 保守性・拡張性

| 項目 | 要件 |
|---|---|
| LLMエンジン追加 | `llm_provider.py` にアダプタを1クラス追加するだけで拡張可能 |
| 会計システム追加 | `mf_api_client.py` と同等のアダプタを追加。`pipeline.py` は変更不要 |
| ストレージ変更 | `drive_client.py` をBox等の同等クラスに差し替え可能な設計 |
| ドキュメント | コードより `sdd.md` / `interface.md` が先。実装はドキュメントに追随 |

---

## 5. コスト

| 項目 | 要件 |
|---|---|
| Google Vision API | 無料枠（1,000件/月）内での運用を前提 |
| LLM API費用 | 個人用途（月30〜60枚）では数十〜数百円以内を想定 |
| MF API | 既存プランに含まれる（追加課金なし） |
| コスト超過時の挙動 | v1では検知しない。将来 Observability で監視を追加 |

---

## 6. 運用制約

| 項目 | 要件 |
|---|---|
| 実行環境 | Python 3.12 / Windows 11（主） / GPU不要 |
| 起動方法 | `python pipeline.py`（手動トリガー）。デーモン化・スケジューラ不要 |
| 設定変更 | `.env` の書き換えのみ。コード変更不要 |
| ログ出力 | ファイル（`logs/`）+ 標準出力。UTF-8（Windows CP932回避済み） |
