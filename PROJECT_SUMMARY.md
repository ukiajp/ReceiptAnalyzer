## 🎯 プロジェクトの哲学

### 「誰でも使えるAI」の実現 - 実用的な日常業務支援を目指して

#### なぜ軽量特化モデルなのか

AI技術の最先端には、大規模モデル（GPT-4、Claude、Llama 70Bなど）が存在します。しかし、実務の現場では「最先端」よりも「確実に動作し、誰でも使える」ことが重要です。

**医療に例えると**:

**教育に例えると**:

- **どちらも価値がある**、そして後者を必要とする人の方が圧倒的に多い

**このプロジェクトが目指すのは「後者」です。**

## 💻 実装の詳細

### 1. OCR部分 (test.py)

#### 主要機能
```python
from paddleocr import PaddleOCR

def extract_text_from_image(image_path):
    ocr = PaddleOCR(lang='japan', use_angle_cls=False, show_log=False)
    result = ocr.ocr(str(image_path), cls=False)
    
    # 位置情報付きデータとして保存
    structured_data = []
    for line in result[0]:
        bbox = line[0]
        text = line[1][0]
        confidence = line[1][1]
        
        structured_data.append({
            'text': text,
            'confidence': confidence,
            'x': int(bbox[0][0]),  # 左上X座標
            'y': int(bbox[0][1]),  # 左上Y座標
            'bbox_full': bbox
        })
    
    # Y座標でソート（上から下）
    structured_data.sort(key=lambda item: (item['y'], item['x']))
    
    return structured_data
```

#### OCR結果の特徴
- 42行のテキストを抽出
- 位置情報（X, Y座標）付き
- 信頼度スコア付き
- **誤字が多い**: 「半」→「¥」、「ギ」→「¥」など

### 2. LLM構造化部分 (test2_simple.py)

#### 重要な設計判断: シンプルアプローチの採用

**失敗したアプローチ**:
- ❌ 位置情報（X/Y座標）を詳細に説明してLLMに渡す
- ❌ 一行ずつの詳細分析を要求（`line_analysis`フィールド）
- ❌ 複雑なJSON構造を要求

**理由**: llama3.2:latestは複雑な指示を理解できず、間違った結果を出力

**成功したアプローチ**:
- ✅ OCRテキストを1行にまとめて渡す
- ✅ シンプルなJSON形式のみ要求
- ✅ 会計システムの目的を明示

#### 最終的なプロンプト構造

```python
prompt = f"""以下はレシートをOCRで読み取ったテキストです。
このデータを会計システムに入力するため、必要な情報を抽出してください。

【OCRテキスト】
{all_text}  # 全テキストを1行にまとめたもの

【会計システムに必要な情報】
以下の情報を抽出し、JSON形式で出力してください：

{{
  "store_name": "店舗名（最も適切なもの）",
  "date": "YYYY-MM-DD",
  "time": "HH:MM",
  "items": [
    {{"name": "商品名", "price": 金額}}
  ],
  "total": 合計金額,
  "tax": 消費税額,
  "payment_method": "支払方法（現金/クレジット/その他）"
}}

【抽出ルール】
- 店舗名: レシート上部にある店名を選択
- 日時: 「YYYY年MM月DD日」と「HH:MM」形式の時刻を探す
- 商品: 商品名と対応する金額をペアにする
- 合計: 「合計」「小計」などのラベルの近くにある最終的な金額
- 消費税: 「消費税」「内消費税」などのラベルの近くの金額
- 支払方法: 「クレジット」「現金」「お預り」などから判断

JSONのみを出力してください。説明や追加情報は不要です。
"""
```

#### JSON抽出ロジック

```python
# LLMの出力から純粋なJSONを抽出
if '```json' in summary:
    json_start = summary.find('```json') + 7
    json_end = summary.find('```', json_start)
    json_str = summary[json_start:json_end].strip()
elif '{' in summary:
    json_start = summary.find('{')
    json_end = summary.rfind('}') + 1
    json_str = summary[json_start:json_end]

parsed_data = json.loads(json_str)
```

## 🎯 重要な知見

### 1. LLMの能力限界を理解する
- **llama3.2:latest**は複雑な構造化タスクが苦手
- 位置情報（座標）を使った推論は期待できない
- シンプルな指示 → 高精度、複雑な指示 → 混乱

### 2. プロンプトエンジニアリングの教訓
```
複雑なプロンプト（失敗）:
- 42行全てを行番号付きで渡す
- X座標<1000は左側、X>1900は右側などの詳細ルール
- line_analysis配列で各行の判定理由を出力
→ LLMが混乱、デタラメな出力

シンプルなプロンプト（成功）:
- 全テキストを1行にまとめる
- 目的を明示（会計システム入力）
- 最小限のJSON構造のみ要求
→ 実用的な精度
```

### 3. OCR誤字への対応は不可避
PaddleOCRの誤字パターン:
- `¥` → `半`、`米`、`ギ`
- `L-` → `レ・`

**対策**: LLMに期待せず、プログラムで確実に修正する必要がある

### 4. 目的駆動の設計が重要
- 「要約してください」→ 曖昧
- 「会計システムに入力するため」→ 明確

def preprocess_ocr_text(texts):
    """LLMに渡す前にOCR誤字を修正"""
    corrected = []
    for text in texts:
        text = text.replace('半', '¥')
        text = text.replace('米', '¥')
        text = text.replace('ギ', '¥')
        text = text.replace('レ・', 'L-')
        corrected.append(text)
    return corrected

   # Before

   # After

   errors='replace'
   ```

   **原因**: Excelなどで既存ファイルが開かれたまま
   
   **解決策**: エラーメッセージ改善（ユーザーに閉じるよう促す）

   except PermissionError:

#### OCR精度の問題発覚

**テスト対象**: FamilyMartレシート（添付画像）

**実際の内容**:
- 店舗: FamilyMart
- 日付: 2024年1月5日 9:38
- 商品: スイートオレンジ&温州みかん ¥108
- 内消費税: ¥8
- 支払: クレジット（VISA 62481）

**OCRの読み取り結果**:
```
FamilyMart 024年 5日 (金) 9:38 999 青NO 4-T39
ソジ&温州 半106車王 キ106 内 ロE ギユ 軽減税率対象商品
2024年 1月 5日 4497 VISト 5248 10
```

**LLMの誤認識結果**:
```json
{
  "store_name": "FamilyMart",
  "date": "2024-01-05",
  "time": "09:38",
  "items": [{"name": "青NO", "price": "999"}],
  "total": "4497",  // 実際は108円
  "tax": "5248",    // 実際は8円
  "payment_method": "クレジット"
}
```

**生成されたCSV（誤り）**:
```csv
1,2024/1/5,消耗品費,事務用品,,FamilyMart,課税仕入 10%,,4497,5248,
未払金,クレジットカード,,,対象外,,4497,0,...
```

**問題の本質**:
- OCRの文字認識精度が低い（"半106" → "半106車王"）
- 2段階処理（OCR → LLM）で誤差が累積
- レイアウト情報の喪失（どこに何が書かれているか）

#### 重要な気づき: Vision LLMアプローチへの方針転換

**現在の方式の限界**:
```
画像 → PaddleOCR → テキスト → LLM → JSON
      ❌誤認識      ❌誤解釈
```
- OCRの誤認識: "¥108" → "半106車王 キ106"
- 文脈の消失: 位置関係が失われる
- 累積誤差: 2段階で誤差が増幅

**Vision LLMアプローチ**:
```
画像 → Vision LLM → JSON
      ✅直接理解（レイアウト含む）
```

**メリット**:
1. OCRステップ不要 → 誤認識を回避
2. レイアウト・文脈を理解（"この金額は合計欄にある"）
3. LoRAで特化可能（レシート専用モデル化）
4. プロジェクト哲学「誰でも使えるAI」に合致
